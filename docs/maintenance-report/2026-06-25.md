## Security Review

| Issue | Severity | Impact | Fix |
| --- | --- | --- | --- |
| `404.html` loaded Bootstrap **3.3.7** from `cdnjs.cloudflare.com` with no Subresource Integrity (SRI) hash. Bootstrap 3.3.7 predates the fix for known XSS advisories in the tooltip/popover/collapse plugins (e.g. CVE-2018-14040, CVE-2018-14041, CVE-2018-14042, CVE-2019-8331), and the unpinned CDN reference is a supply-chain risk (if cdnjs ever serves different content for that path, the page would load it unverified). | Medium | The 404 page only uses basic grid classes today, so the vulnerable plugins aren't actively invoked, but the exposure existed for any future use of tooltips/popovers on that page, plus the unpinned third-party CDN load itself. | Replaced with the site's own vendored, current Bootstrap 4.6.0 (`Content/bootstrap.min.css`); updated the one Bootstrap-3-only class (`col-sm-offset-1` → `offset-sm-1`) so the layout is unchanged. No more external Bootstrap load on this page. |
| `debug.log` (a local Chromium internal settings/error log) was committed to the repository root. | Low | No secrets in the file, but it's irrelevant local-machine noise that shouldn't be in source control. | Removed from git and added to `.gitignore`. |

## Testing

- Added `scripts/check-local-links.js`, which scans `index.html` and `404.html` for local `href`/`src` references (ignoring HTML comments and external/`mailto:`/`tel:`/`data:` URLs) and fails if any referenced local file is missing.
- Ran it against the repository before and after changes:
  - **Before fixes:** flagged 3 references, all of which were inside HTML comments (`index2.html`, `Content/File/resume_reo.pdf`, `Content/Hola/js/music.js`) — i.e. intentionally disabled features, not live bugs. The checker was adjusted to strip comments so it doesn't false-positive on disabled code.
  - **After fixes:** `OK: checked 2 HTML file(s), all local references resolve.`
- No other automated test suite exists for this static site; there is no application logic to unit test beyond the link checker itself.

## CI/CD

- No CI/CD existed previously.
- Added `.github/workflows/check.yml`: runs on push to `main` and on every pull request, and runs `node scripts/check-local-links.js`. This blocks merges when a local asset reference is broken.
- Deployment itself is unchanged — GitHub Pages continues to deploy directly from `main` (repository Settings → Pages), which this workflow does not touch.
- A blocking strict-HTML-validator gate (`html-validate` with default rules) was evaluated and rejected: against the existing template it produced 227 errors (inline styles, self-closing void elements, legacy IE conditional comments, accessibility rules), all pre-existing in the vendored "Hola" template across years of history. Fixing those would be a large, unrelated rewrite — out of scope for this pass — so a narrower, zero-false-positive check was used instead.
