### Security review (dependencies)

No CVEs were identified against jQuery 3.7.0/3.7.1 or Bootstrap 4.6.0/4.6.2 in public advisories checked. Severity for all listed items: **No known vulnerability**. The jQuery and Bootstrap bumps were treated as routine patch-level hygiene, not vulnerability remediation.

**Flagged risk (not fixed this pass)**: `404.html` still loads Bootstrap **3.3.7** from `cdnjs.cloudflare.com`. Bootstrap 3.x is end-of-life and had historical XSS advisories (in `data-target`/`tooltip`/affix handling) fixed in later 3.x/4.x releases. Since this page has no user input and only renders a static "page not found" message, the practical exploitability is very low, but it is inconsistent with the rest of the site (which now uses jQuery 3.7.1 and no longer ships Bootstrap 4 at all) and should be revisited — either drop the Bootstrap dependency from `404.html` entirely (it's barely used) or move it to a currently-maintained Bootstrap major, pinned by version rather than a mutable CDN path with no Subresource Integrity (SRI) hash.

## Code Quality

- **Removed dead code**: `Content/bootstrap.css`, `Content/bootstrap.min.css`, `Content/bootstrap-grid.css`, `Content/bootstrap-grid.min.css`, `Content/bootstrap-reboot.css`, `Content/bootstrap-reboot.min.css` (plus their `.map` files) and `Content/Site.css` — none of these were referenced by any `<link>`/`<script>` tag anywhere in the repository. They were confirmed unreferenced via a full-repo search before removal.
- **Fixed a real markup bug**: in `index.html`, the `<section id="projects">` block opened a `<div data-aos="zoom-in">` (line 397) that was never closed, causing every element after it (testimonials, contact, closing `</body>`/`</html>`) to be structurally unbalanced per the HTML5 spec. Browsers silently auto-recover from this via error-correction, so there was no visible symptom, but it is fragile and was flagged by HTML validation. Added the missing `</div>`.
- **Fixed 404.html document structure**: the page's `<style>` block was placed *before* `<head>` opened, so browsers implicitly opened a `<head>`, and the literal `<head>` tag that followed was then illegally nested inside it. Moved the `<style>` block inside `<head>`, after the `<meta>`/`<title>`/`<link>` tags, with no visual or behavioral change.
- **Fixed unencoded ampersands**: replaced four instances of a raw `&` in visible text content (e.g. "HTML & CSS") with `&amp;`, per the HTML spec.
- **Stripped a UTF-8 BOM** from the start of `index.html` (harmless in modern browsers but nonstandard and flagged by validation).
- **Not changed** (documented as accepted technical debt, out of scope for this pass): several elements intentionally nest block-level `<div>`s inside `<h1>`/`<h3>` headings (e.g. the `flipletter`/`movingletter` hero text) and use empty headings as JS injection targets for the site's text-animation libraries. These trigger HTML-validator "content model" warnings but are working, intentional integration points for third-party animation plugins — rewriting them risks breaking the animations and was judged not worth the risk for a cosmetic linting improvement. These rules were explicitly disabled in `.htmlvalidate.json` with the reasoning documented there.
- **Not changed**: the large hand-concatenated `Content/Hola/js/plugins.js` bundle (Waypoints, Parallax.js, PhotoSwipe v4.1.2, etc.). Upgrading any one embedded library requires re-bundling by hand and carries meaningful regression risk for a personal site with no test coverage of its JS interactions; flagged as future work rather than attempted blind.
